## 🌍 Обзор

На текущем этапе AgInTi работает как пайплайн жизненного цикла README и локализации, а не как runtime-приложение. Корневой `README.md` является каноническим источником, а локализованные варианты в `i18n/` синхронизируются с этой канонической структурой.

Философия проекта выражена явно и применяется на практике, а не для оформления. Каждое обновление README должно соответствовать всем трём принципам:

1. **Sear creation tools**: намеренно острые процессы создания, которые формируют высокосигнальную и практичную документацию на основе ограниченных данных репозитория.
2. **Self-healing tools**: механизмы самовосстановления, которые исправляют дрейф, убирают дублирование и возвращают структурную согласованность.
3. **Chain of prompt tools**: этапные и отслеживаемые потоки промптов, сохраняющие связь контекста и результата между запусками пайплайна.

Репозиторий сохраняет содержательные исторические материалы через инкрементальные обновления, при этом бережно сохраняя ссылки, команды и метаданные поддержки.

### Философия в двух словах

| Принцип | Намерение | Практический результат |
| --- | --- | --- |
| **Sear creation tools** | Формировать высокосигнальную документацию на базе ограниченных данных. | Разделы остаются практичными, конкретными и привязанными к репозиторию. |
| **Self-healing tools** | Исправлять дрейф, дублирование и устаревшую структуру. | Канонический и локализованные README остаются синхронизированными и чистыми. |
| **Chain of prompt tools** | Делать этапы генерации явными и прослеживаемыми. | Артефакты пайплайна сохраняют воспроизводимый контекст и передачу между этапами. |

## ✨ Возможности

- README-first стратегия документации с каноническим корневым документом.
- Многоязычная синхронизация 10 вариантов README в i18n.
- Пайплайн-ориентированное ведение документации через артефакты `.auto-readme-work/<run-id>/`.
- Инварианты «один баннер» и «одна панель поддержки», предотвращающие дубли визуальных блоков.
- Дисциплина инкрементальных обновлений, сохраняющая значимую техническую историю.

### Соответствие принципов и возможностей

| Базовый принцип | Как проявляется в текущих возможностях |
| --- | --- |
| **Sear creation tools** | Точное написание README на основе фактов репозитория и стабильных шаблонов разделов. |
| **Self-healing tools** | Проверки дедупликации повторяющихся баннеров/блоков поддержки, устаревших ссылок и структурного дрейфа. |
| **Chain of prompt tools** | Цепочка артефактов конкретного запуска (`pipeline-context`, шаблоны навигации, план перевода) для воспроизводимого результата. |

## 🏗️ Архитектура

На этом этапе под архитектурой понимается архитектура документационного пайплайна, а не runtime-архитектура приложения.

### Поток пайплайна

1. Контекст конкретного запуска фиксируется в `.auto-readme-work/<run-id>/pipeline-context.md`.
2. Генерируются шаблоны языковой навигации (`language-nav-root.md` и `language-nav-i18n.md`).
3. Канонический README обновляется инкрементально в `README.md`.
4. Локализованные README в `i18n/` выравниваются по канонической структуре.
5. Проверки структурного качества обеспечивают дедупликацию, согласованность ссылок и сохранение технических деталей.

### Основные принципы в архитектуре

- **Sear creation tools**: применяются при создании контента, чтобы разделы оставались конкретными, полными и точными относительно репозитория.
- **Self-healing tools**: применяются при валидации для удаления дубликатов, исправления устаревших ссылок на запуски и восстановления структурного паритета.
- **Chain of prompt tools**: применяются ко всем артефактам, чтобы каждый этап генерации оставался явным и проверяемым.

### Контроль принципов по этапам пайплайна

| Этап | Sear creation tools | Self-healing tools | Chain of prompt tools |
| --- | --- | --- | --- |
| Сбор контекста | Определяют чёткие ограничения генерации. | Рано выявляют отсутствующие/невалидные входные данные. | Сохраняют исходный промпт и метаданные запуска. |
| Подготовка канонического текста | Формируют полные разделы README на основе данных репозитория. | Предотвращают регрессии и случайную потерю контента. | Сохраняют связь выходов этапа с предыдущими артефактами. |
| Выравнивание i18n | Поддерживают структурный и технический паритет между локалями. | Исправляют дрейф между корневым и i18n-файлами. | Переносят замысел канонической версии в каждый локализованный вариант. |
| Финальная проверка | Обеспечивают читаемость и сохранность деталей. | Удаляют дубликаты баннера/поддержки и устаревшие ссылки. | Оставляют проверяемый след артефактов по запуску. |

## ▶️ Использование

Текущее использование сосредоточено на сопровождении документации и многоязычной синхронизации.

### Типовые команды для проверки

```bash
ls -la
ls -la .auto-readme-work/20260302_124338
ls -la i18n
```

### Процесс синхронизации канонического README

1. Прочитайте `.auto-readme-work/20260302_124338/pipeline-context.md`.
2. Проверьте шаблоны языкового селектора из `language-nav-root.md` и `language-nav-i18n.md`.
3. Инкрементально обновите `README.md` как источник истины.
4. Выравнивайте файлы `i18n/README.*.md` по той же структуре и ключевым техническим деталям.
5. Убедитесь, что присутствует ровно один баннер и ровно один блок поддержки.

## ⚙️ Конфигурация

Runtime-конфигурация пока отсутствует. Поведение документации определяется артефактами репозитория.

- `pipeline-context.md`: цели и ограничения запуска.
- `repo-structure-analysis.md`: фактический снимок и пробелы.
- `language-nav-root.md` и `language-nav-i18n.md`: согласованность навигации.
- `translation-plan.txt`: цели локалей и соответствия.

## 🛠️ Заметки по разработке

- Сохраняйте содержательные разделы и ссылки из истории канонического README.
- Предпочитайте инкрементальные правки вместо разрушительных переписываний.
- Сохраняйте только один баннер и только один блок поддержки.
- Поддерживайте синхронизацию структуры корневого и i18n README.
- Явно фиксируйте предположения, когда детали runtime или инфраструктуры неизвестны.
- Применяйте триаду философии как рабочие ограничители:
  - **Sear creation tools** для высокосигнального написания.
  - **Self-healing tools** для восстановления согласованности.
  - **Chain of prompt tools** для воспроизводимой передачи контекста между этапами пайплайна.

### Строки языкового селектора отличаются между файлами

Используйте канонические шаблоны в:

- `.auto-readme-work/20260302_124338/language-nav-root.md`
- `.auto-readme-work/20260302_124338/language-nav-i18n.md`

```bash
git fetch origin
git pull --ff-only
```

### Я хочу добавить runtime-инструкции

Добавляйте инструкции по сборке/запуску только после добавления конкретных манифестов (например: `package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`) и подтверждения путей в этом репозитории.

## 🗺️ Дорожная карта

1. Усилить **sear creation tools** за счёт стандартизированных шаблонов подготовки README, порогов качества разделов и более прозрачных проверок «доказательства → результат».
2. Расширить **self-healing tools** автоматическими проверками дубликатов блоков, дрейфа локалей, сломанных внутренних якорей и устаревших ссылок на запуски.
3. Формализовать **chain of prompt tools** между этапами запуска для воспроизводимых трасс контекста, генерации, перевода и верификации.
4. Добавить однокомандный поток сопровождения документации после появления скриптов в репозитории.
5. Добавить проверки CI для качества markdown, целостности ссылок и паритета структуры i18n.
6. Ввести конкретные runtime-компоненты после добавления исходных манифестов и entrypoint-ов.
7. Принять стабильное решение по лицензии и добавить отдельный файл лицензии.

### Дорожная карта по фокусу принципов

| Область фокуса | Ближайшая цель |
| --- | --- |
| **Sear creation tools** | Улучшить шаблоны написания и промпты разделов с опорой на данные репозитория. |
| **Self-healing tools** | Автоматизировать обнаружение дубликатов, проверку устаревших якорей и исправление дрейфа локалей. |
| **Chain of prompt tools** | Стандартизировать контракты артефактов этапов запуска для воспроизводимых многоязычных результатов. |
