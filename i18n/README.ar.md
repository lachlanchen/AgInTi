## 🌍 نظرة عامة

يعمل AgInTi حاليًا كخط دورة حياة README وخط توطين، وليس كتطبيق وقت تشغيل. ويعد `README.md` في الجذر هو المصدر المرجعي، بينما تتم مزامنة النسخ المحلية في `i18n/` انطلاقًا من هذا الهيكل المرجعي.

فلسفة المشروع واضحة وتشغيلية وليست شكلية. يجب أن يحقق كل تحديث لـ README المبادئ الثلاثة كلها:

1. **Sear creation tools**: سير إنشاء حاد ومقصود ينتج توثيقًا عمليًا عالي الإشارة انطلاقًا من أدلة مستودع مقيدة.
2. **Self-healing tools**: آليات إصلاحية تعالج الانحراف، وتزيل التكرار، وتعيد الاتساق البنيوي.
3. **Chain of prompt tools**: تدفقات prompts مرحلية وقابلة للتتبع تحفظ سلسلة السياق إلى المخرجات عبر تشغيلات خط الأنابيب.

يحافظ هذا المستودع على المحتوى التاريخي الجوهري عبر تحديثات تدريجية مع صون الروابط والأوامر وبيانات الدعم.

### الفلسفة بنظرة سريعة

| المبدأ | الهدف | النتيجة التشغيلية |
| --- | --- | --- |
| **Sear creation tools** | إنتاج توثيق عالي الإشارة من أدلة مقيدة. | تبقى الأقسام عملية ومحددة ومرتكزة على المستودع. |
| **Self-healing tools** | إصلاح الانحراف والتكرار والبنية المتقادمة. | تبقى ملفات README المرجعية والمحلية متسقة ونظيفة. |
| **Chain of prompt tools** | إبقاء مراحل التوليد صريحة وقابلة للتتبع. | تحفظ مخرجات خط الأنابيب سياقًا قابلًا لإعادة الإنتاج وتسليمًا واضحًا. |

## 🏗️ المعمارية

في هذه المرحلة، تعني المعمارية معمارية خط أنابيب التوثيق، وليست معمارية تطبيق وقت تشغيل.

### تدفق خط الأنابيب

1. يُسجل سياق خاص بكل تشغيل في `.auto-readme-work/<run-id>/pipeline-context.md`.
2. تُولّد قوالب تنقل اللغة (`language-nav-root.md` و`language-nav-i18n.md`).
3. يُحدّث محتوى README المرجعي تدريجيًا في `README.md`.
4. تُحاذى ملفات README المحلية في `i18n/` مع البنية المرجعية.
5. تفرض فحوص الجودة البنيوية إزالة التكرار، واتساق الروابط، والحفاظ على التفاصيل التقنية.

### المبادئ الأساسية داخل المعمارية

- **Sear creation tools**: تُطبق أثناء بناء المحتوى لإبقاء الأقسام ملموسة ومكتملة ودقيقة بالنسبة للمستودع.
- **Self-healing tools**: تُطبق أثناء التحقق لإزالة الكتل المكررة، وإصلاح مراجع التشغيل المتقادمة، واستعادة التكافؤ البنيوي.
- **Chain of prompt tools**: تُطبق عبر المخرجات بحيث تبقى كل مرحلة توليد صريحة وقابلة للتدقيق.

### نقاط تحقق المبادئ حسب مرحلة خط الأنابيب

| المرحلة | Sear creation tools | Self-healing tools | Chain of prompt tools |
| --- | --- | --- | --- |
| التقاط السياق | تحديد قيود توليد حادة وواضحة. | الإشارة مبكرًا إلى المدخلات الناقصة/غير الصالحة. | حفظ prompt المصدر وبيانات التشغيل الوصفية. |
| الصياغة المرجعية | بناء أقسام README مكتملة من أدلة المستودع. | منع التراجعات وفقدان المحتوى بالخطأ. | إبقاء مخرجات المراحل مرتبطة بالمخرجات السابقة. |
| مواءمة i18n | الحفاظ على البنية والتكافؤ التقني عبر اللغات. | تصحيح الانحراف بين الجذر وملفات i18n. | نقل القصد المرجعي إلى كل نسخة محلية. |
| التحقق النهائي | فرض قابلية القراءة والحفاظ على التفاصيل. | إزالة البانر/لوحة الدعم المكررين والمراجع المتقادمة. | ترك سلسلة مخرجات قابلة للتدقيق لكل تشغيل. |

## 🗺️ خارطة الطريق

1. تعزيز **sear creation tools** عبر قوالب صياغة README معيارية، وبوابات جودة للأقسام، وفحوص أوضح من الأدلة إلى المخرجات.
2. توسيع **self-healing tools** عبر فحوص آلية لاكتشاف الكتل المكررة، وانحراف اللغات، والمراسي الداخلية المكسورة، ومراجع التشغيل المتقادمة.
3. إضفاء طابع رسمي على **chain of prompt tools** عبر مراحل التشغيل لتحقيق سجلات قابلة لإعادة إنتاج السياق والتوليد والترجمة والتحقق.
4. إضافة تدفق صيانة توثيق بأمر واحد عند إدخال سكربتات للمستودع.
5. إضافة فحوص CI لجودة markdown وسلامة الروابط وتكافؤ بنية i18n.
6. إدخال مكونات وقت تشغيل فعلية عند إضافة manifests ونقاط دخول للمصدر.
7. نشر قرار ترخيص مستقر وإضافة ملف ترخيص مستقل.

### خارطة طريق حسب تركيز المبدأ

| مجال التركيز | الهدف القريب |
| --- | --- |
| **Sear creation tools** | تحسين قوالب الصياغة وprompts الأقسام المعتمدة على الأدلة. |
| **Self-healing tools** | أتمتة اكتشاف التكرار، وفحوص المراسي المتقادمة، وإصلاح انحراف اللغات. |
| **Chain of prompt tools** | توحيد عقود مخرجات مراحل التشغيل لإنتاج متعدد اللغات قابل لإعادة الإنتاج. |
